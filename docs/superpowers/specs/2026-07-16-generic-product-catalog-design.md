# 通用商品目录与视频生成改造 — 设计文档

## 1. 背景与目标

当前系统围绕香水商品设计，产品模型、前端表单、API、Agent 提示词和测试均包含前调、中调、后调等固定字段。本次改造将系统扩展为适用于各种商品的短视频生成平台，同时保留用户按自身业务定义品类和属性的能力。

目标如下：

- 每个用户独立维护自己的品类、属性模板和商品，不设置平台级统一商品目录。
- 新用户注册后异步初始化一套可自由修改和删除的样例品类与样例商品。
- 商品使用通用基础字段，并通过所属品类的模板扩展动态属性。
- 商品主图在最终保存时必填；用户未上传时，可使用现有 AI 图片生成能力生成候选图，确认后作为主图。
- 视频任务创建时保存完整商品快照，后续商品或品类修改不影响已有任务。
- 三类视频 Agent 从香水专用提示词改为通用商品提示词。
- API 契约统一维护在 `api/openapi.yaml`，数据库完整 DDL 统一维护在 `db/schema.sql`。

现有数据均为开发数据，可以清空重建。本次不提供旧香水字段的线上数据迁移或 API 兼容层。

## 2. 方案选择

采用“关系型品类模板 + JSONB 商品属性”的混合模型：

- `categories` 保存用户私有品类。
- `category_attributes` 以关系记录保存字段定义、类型、选项、必填规则和顺序。
- `products` 以固定列保存通用字段，并以 `attributes JSONB` 保存动态属性值。
- `video_tasks.product_snapshot JSONB` 保存任务创建时的商品快照。

该方案比全 JSONB 模型具备更明确的模板约束和更新边界，又避免完全 EAV 模型带来的复杂查询与接口成本。

## 3. 多租户边界

品类、商品、AI 主图候选图均归属于用户。服务端只能从当前认证用户获得 `user_id`，不接受客户端指定资源所属用户。

所有读取和修改操作必须同时使用资源 ID 与当前用户 ID 查询。跨用户资源应表现为不存在，防止泄露其他用户的数据。数据库为常用的用户过滤字段建立索引，并通过唯一约束保证同一用户下品类名称唯一。

视频任务继续归属于创建用户。创建任务时必须验证所选商品属于当前用户。

## 4. 领域模型

### 4.1 品类

`categories` 包含：

- `id`
- `user_id`
- `name`
- `description`
- `template_version`
- `created_at`
- `updated_at`

同一用户下品类名称唯一。品类被任一商品引用时禁止删除，用户需先迁移或删除关联商品。服务层提供易理解的冲突信息，数据库外键使用限制删除作为最终保护。

`template_version` 在属性模板发生变化时递增，用于表单并发控制和商品提交校验。

### 4.2 品类属性模板

`category_attributes` 包含：

- `id`
- `category_id`
- `key`：稳定的机器字段名，在品类内唯一
- `label`：展示名称
- `type`：`text`、`number`、`single_select`、`multi_select`、`boolean`
- `required`
- `options JSONB`：仅单选和多选使用
- `sort_order`
- `created_at`
- `updated_at`

后端校验属性定义：选择类型必须包含非空且不重复的选项；非选择类型不得携带选项。

模板修改不批量重写已有商品。删除模板字段后，已有商品 JSONB 中的旧值可暂时保留为历史数据，但不再出现在编辑表单、任务快照或新生成内容中。商品再次保存时，仅接受当前模板定义的属性。

### 4.3 商品

`products` 包含：

- `id`
- `user_id`
- `category_id`
- `name`
- `description`
- `selling_points JSONB`
- `scenarios JSONB`
- `main_image_url`
- `main_image_source`：`upload` 或 `ai`
- `attributes JSONB`
- `category_template_version`
- `created_at`
- `updated_at`

保存商品时，后端根据当前品类模板执行完整校验：

- 所有模板必填属性均有值。
- 数字和布尔类型正确。
- 单选值属于当前选项。
- 多选值均属于当前选项且不重复。
- 不接受当前模板未定义的客户端属性。
- 客户端提交的模板版本必须等于当前版本，否则返回冲突，要求刷新表单。
- `main_image_url` 最终必须存在。

### 4.4 视频任务商品快照

`video_tasks` 新增 `product_snapshot JSONB`。快照至少包含：

- 商品 ID、名称、描述、卖点、适用场景和主图。
- 品类 ID、品类名称和描述。
- 当前有效属性的 key、label、type 和规范化后的值。
- 快照版本和创建时间。

Agent、任务详情及任务重试均只读取快照，不在运行过程中重新读取商品。`video_tasks.product_id` 改为可空外键并使用 `ON DELETE SET NULL`，从而允许删除商品而不破坏已有任务。

## 5. 注册后样例初始化

注册模块与样例复制代码通过领域事件隔离：

1. 注册事务创建用户，同时写入 `user.registered` Outbox 事件。
2. 独立的 `SampleCatalogInitializer` 消费该事件。
3. 初始化器依据版本化的样例定义创建样例品类、属性模板和样例商品。
4. 初始化状态记录为 `pending`、`completed` 或 `failed`，并保存失败原因和重试信息。

当前阶段可以使用进程内 Outbox 分发器执行；后续替换为消息队列发布者和消费者时，不修改注册服务及初始化领域接口。

初始化必须幂等。以用户 ID、样例版本和稳定样例 key 建立唯一约束，重复消费不会产生重复数据。初始化失败不回滚已成功的注册，消费者按退避策略重试。初始化过程中商品页面显示“正在准备样例数据”，但用户仍可自行创建品类和商品。

样例数据创建后即为普通用户私有数据，不与样例定义保持同步，用户可自由修改和删除。

## 6. 商品与品类维护页面

### 6.1 商品管理

新增“商品管理”一级页面，包含：

- 商品列表、名称搜索和品类筛选。
- 商品主图、名称、品类、核心卖点摘要和更新时间展示。
- 新增、编辑和删除操作。
- 删除前展示关联视频任务数量；删除商品不删除已有任务。

新增与编辑共用商品表单。选择品类后动态加载属性模板。切换品类时保留通用字段，但在用户确认后清除旧品类动态属性。

表单支持上传、预览、替换和移除主图。没有主图时不能最终保存商品，用户可以转入 AI 生成主图流程。

### 6.2 AI 商品主图

AI 主图采用两阶段确认：

1. 用户填写商品草稿并请求生成。
2. 后端以商品名称、品类、描述、卖点、场景和动态属性构建通用生图提示词。
3. 调用现有图片生成能力，创建归属于当前用户的临时候选图。
4. 前端预览候选图，允许重新生成、改为上传或确认使用。
5. 用户确认后，保存商品时将候选图转为正式商品资产。

候选图包含过期时间和使用状态。保存商品时必须验证候选图属于当前用户、尚未过期且未被其他商品使用。未确认的过期资产由后台任务清理。

### 6.3 品类管理

新增“品类管理”页面，展示品类、属性数量和关联商品数量。用户可维护品类基础信息，并通过属性编辑器配置字段名称、类型、必填状态、顺序和选择项。

被商品引用的品类不显示可执行的删除操作，或在执行时返回明确的阻止原因。模板更新使用版本号进行乐观锁控制。

## 7. API 契约

以下接口及请求、响应、错误结构必须完整定义在 `api/openapi.yaml`：

- `GET /categories`
- `POST /categories`
- `GET /categories/{category_id}`
- `PUT /categories/{category_id}`
- `DELETE /categories/{category_id}`
- `GET /products`
- `POST /products`
- `GET /products/{product_id}`
- `PUT /products/{product_id}`
- `DELETE /products/{product_id}`
- `POST /products/main-image/generate`
- `GET /initialization-status`

品类属性模板随品类详情返回，并可随品类创建和更新整体提交。这样一次请求即可原子更新模板，避免多个字段级接口造成部分成功。

主图生成接口接收未保存的商品草稿，返回 `candidate_id`、预览地址和过期时间。商品创建或更新接口接受已确认候选图 ID，服务端完成资产转正。

商品列表支持名称、品类、分页和排序参数。所有资源接口均执行当前用户隔离。

## 8. Agent 与任务数据流

引入统一的 `ProductContext`，由任务快照构造，包含：

- 名称、品类、描述、卖点、场景和主图。
- 已按属性模板顺序格式化的动态属性。
- 适合 LLM 使用的结构化对象和稳定文本表示。

三类 Agent 的调整如下：

- `promo`：采用吸引注意、需求或场景、核心卖点、属性佐证、行动引导的通用商品广告结构。
- `viral`：将 `ProductContext` 套入参考视频的结构、镜头和风格，不包含任何固定品类描述。
- `personify`：依据商品品类、外观、卖点和属性设计拟人角色，并生成第一人称文案。

生图和视频提示词使用商品实际品类、外观和主图参考，移除香水瓶、香调和香水广告风格等硬编码。香水作为普通用户品类存在；用户定义前调、中调和后调属性后，仍能通过通用上下文获得原有表达能力。

任务数据流为：

1. 校验商品归属与完整性。
2. 在任务创建事务内生成并写入商品快照。
3. 后台任务只从快照构造 `ProductContext`。
4. Agent 生成脚本、图片、视频和配音。
5. 商品被修改或删除时，已有任务继续使用原快照完成审核、重试和渲染。

## 9. 数据库与实现边界

最终数据库结构完整维护在 `db/schema.sql`，包括表、索引、唯一约束、检查约束、外键和更新时间触发器。DDL 中应调整定义顺序，确保更新时间函数在引用它的触发器之前创建。

需要移除：

- `products.top_note`
- `products.middle_note`
- `products.base_note`

需要新增或调整：

- 用户与品类、商品、主图候选图的归属关系。
- 品类及品类属性模板表。
- 商品通用字段和动态属性。
- Outbox、初始化状态及幂等记录。
- 视频任务的商品快照与可空商品外键。

应用模型、Pydantic Schema、前端类型和 `api/openapi.yaml` 必须与 DDL 的字段语义保持一致。

## 10. 错误处理

- 跨用户资源访问统一按资源不存在处理。
- 品类被引用时删除返回冲突，并包含关联商品数量。
- 模板版本过期时返回冲突和当前版本。
- 属性校验错误按字段返回，便于前端定位。
- AI 生图失败时保留表单草稿，允许重试或上传图片。
- 商品保存失败时不立即销毁仍有效的候选图。
- 候选图过期、已使用或归属不匹配时返回明确错误。
- AI 生成主图继续使用现有安全策略；失败不能绕过主图必填规则。
- Outbox 消费失败记录可观测错误并重试，不影响用户登录或手动创建目录数据。

## 11. 测试策略

### 11.1 模型与服务测试

- 同一用户下品类名称唯一，不同用户可重名。
- 五种属性类型、必填规则、选项约束和未知字段拒绝。
- 模板版本冲突。
- 被商品引用的品类禁止删除。
- 商品删除后任务快照仍完整可用。

### 11.2 API 测试

- 品类和商品 CRUD、搜索、筛选与分页。
- 所有资源的跨用户读取和写入拒绝。
- AI 主图候选图生成、确认、过期、重复使用和归属验证。
- 创建视频任务时快照内容完整且只包含当前有效属性。

### 11.3 初始化测试

- 注册事务写入 Outbox 事件。
- 重复消费不产生重复样例。
- 部分失败后重试可收敛到完整状态。
- 不同用户的样例数据完全隔离。
- 初始化失败时用户仍可自行创建品类和商品。

### 11.4 Agent 与流程测试

- 使用香水、电子产品和食品等不同商品验证提示词无固定品类措辞。
- 三类视频任务均从快照构造上下文。
- 商品更新后已有任务输出仍使用原快照。
- 商品删除后已有任务可继续完成生成、审核和渲染。

### 11.5 前端测试

- 品类模板创建、编辑、排序和删除限制。
- 根据模板动态渲染商品字段并显示字段级错误。
- 切换品类时的确认和动态属性清理。
- 主图上传、AI 生成、重新生成和确认。
- 初始化进行中和失败状态展示。

## 12. 实施与隔离约束

当前工作目录存在其他任务的未提交修改。实施前应使用独立 Git worktree 隔离本次改造，避免覆盖共享工作树中的变更。实施必须以更新 `api/openapi.yaml` 和 `db/schema.sql` 为显式任务，并在各层变更后验证契约、模型和前端类型的一致性。

本设计的范围不包括平台级品类市场、跨用户共享品类、复杂单位与日期属性、自动升级用户样例数据，以及对旧香水开发数据的迁移。
